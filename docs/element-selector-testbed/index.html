Element Selector Testbed — v1

Two strategies (explicitTrackingAttribute + stableId) plus a hardened fallback that filters autogenerated ids and library state classes through the pattern packs before sibling disambiguation.
How v1 differs from today's cssPath: Today's algorithm uses any id as an anchor (including autogenerated ones like React's :r5:) and includes every class on an element when disambiguating among siblings (including library state classes like swiper-slide-active). v1 filters both through pattern packs — autogenerated ids never anchor a selector, and unstable classes never appear in sibling disambiguation. Classes are never used as primary anchors in v1; that role is reserved for explicit tracking attributes (data-amp-track-id) and stable ids.

How v1 works

When autocapture captures a click, v1 runs a small algorithm to produce a CSS selector string that goes into the [Amplitude] Element Path event property. The algorithm has three passes; the first that produces a uniquely-matching selector wins.

Pass 1
Try strategies on the target itself

For the clicked element, run each strategy in order. Stop on the first one that produces a selector matching exactly one element in the document.

  • explicitTrackingAttribute Reads data-amp-track-id="value". If present and non-empty, returns [data-amp-track-id="value"]. Empty value (data-amp-track-id="") is a suppression signal — strategy returns null and other strategies / the fallback skip this element's id.
  • stableId Reads the element's id. If the id is present, non-suppressed, and doesn't match any pattern in the autogenerated-id pack (React :r5:, Radix, MUI, hashes, 4+ digit runs, etc.), returns tag#id. Otherwise null.

Hits here mean clean, short selectors like [data-amp-track-id="signup"] or button#checkout.

Pass 2
Walk up the ancestor chain

If pass 1 produces nothing unique, walk from the target's parent up toward <html>. At each ancestor, run the same strategy chain. The first ancestor where any strategy produces a unique selector becomes the anchor.

The selector is built as anchor > descent, where descent describes the target relative to the anchor using tag:nth-of-type(n) at each step. This matches ContentSquare's "position within identical markers" convention.

Example outputs: main#checkout > section:nth-of-type(1) > button:nth-of-type(1), or [data-amp-track-id="login-form"] > button:nth-of-type(1) when a container is customer-tagged.

Pass 3
Hardened fallback (fallback-css-path)

If the walk reaches <html> without finding a unique anchor, fall back to a structural path from target to root. Two differences from today's legacy cssPath:

  • Id filtering: the same autogenerated-id pack used by stableId is applied at every step of the walk. An autogenerated id encountered mid-walk doesn't halt the walker — it's treated as if the element had no id.
  • Class filtering: when sibling disambiguation forces classes into the selector, each class is first filtered through the unstable-class pack (Tailwind utilities, CSS-in-JS hashes, library state classes). Only surviving classes are added. If filtering removes every discriminating class, the walker emits :nth-child(N).

Example output: html > body > div > main > … > div:nth-child(2) > a with no autogenerated ids or state classes anywhere.

Why these specific design choices

Classes never anchor

The class strategy is intentionally not in v1. Class names are runtime state — a transient hover, focus, or loading class can pollute the anchor and produce a selector that's unique now but breaks the moment the state changes. v1 only filters classes inside the fallback's sibling disambiguation; classes never become primary anchors.

Two regex packs, tunable per customer

Both the autogenerated-id pack and the unstable-class pack ship with sensible defaults. Customers can add new patterns or remove ones that don't fit their app via remote config — the defaults are surfaced so customers can see exactly what's being filtered and adjust as needed. New library state-class conventions can be picked up without an SDK release.

Walk to <html> by default

The walk has no depth bound by default — DOM depth is finite, traversal is cycle-free, per-event cost is microseconds. An optional maxAncestorWalkDepth in remote config lets operators throttle for pathological pages.

Customer overrides take priority

data-amp-track-id is consulted first at every level. Customers can tag containers to force anchoring on their semantic markers, or suppress an inner dynamic id with an empty value. Same ergonomics as ContentSquare's data-cs-override-id.

What stays unchanged

  • The event-property name ([Amplitude] Element Path) and wire format. v1 changes the value, not the property.
  • The walk's terminal condition — both algorithms halt at a stable id or at <html>.
  • The CSS selector separator (>).
  • Every other event property (Element Tag, Element Text, Element ID, Element Class, etc.) — computed exactly as today.
  • The kill-switch path. When v1 is disabled per-customer via remote config, autocapture falls back to today's legacy cssPath byte-for-byte.
