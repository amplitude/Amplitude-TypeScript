# @amplitude/element-selector

Shared element-selector algorithm consumed by the autocapture SDK plugin, the app.amplitude.com tagging UI, and the Chrome extension visual tagger. Replaces the copy-paste `cssPath` pattern that previously lived independently in each consumer.

## Why a separate package

The v1 selector algorithm is consumed by three surfaces that today each copy their own variant of the legacy `cssPath` logic:

1. **`@amplitude/plugin-autocapture-browser`** — used at runtime in customers' websites to generate selectors for captured click events.
2. **app.amplitude.com tagging UI** — used in the Amplitude product to let customers tag zones on their pages. Selectors saved here must match selectors emitted by the SDK on the same DOM.
3. **Chrome extension visual tagger** — same purpose as the tagging UI but injected on the customer's live page.

If these three surfaces drift apart (different filtering logic, different patterns), saved zones stop matching captured events. Extracting the algorithm into one shared package keeps them in lockstep by construction.

## Install

```bash
pnpm add @amplitude/element-selector
```

## Usage

API surface lands in ENG-002 and onwards. Once shipped:

```ts
import { createSelectorEngine } from '@amplitude/element-selector';

// SDK pattern: subscribe to remote config
const engine = createSelectorEngine({
  remoteConfigClient: amplitudeConfig.remoteConfigClient,
  localOptions: pluginOptions.elementSelector,
});

const selector = engine.generate(clickedElement);
```

For the three consumer integration patterns (SDK / dashboard / Chrome extension), see the "Consumer integration" section of the design doc.
